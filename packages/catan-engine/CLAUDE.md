# catan-engine library

## Layout

The package is organised into three subpackages by role, layered so imports only
ever point "down" (`env` ‚Üí `mechanics` ‚Üí `board`; no cycles):

- `board/` ‚Äî **data**: the game's primitives (enums + constant tables) and its
  static/dynamic representation.
- `mechanics/` ‚Äî **logic**: the traceable single-game rule helpers and the
  action layer that composes them.
- `env/` ‚Äî **RL surface**: the batched env and the single-game PettingZoo-AEC
  wrapper.

## Source files

Keep this section up to date when adding, removing, or significantly changing a file.

### `board/` ‚Äî primitives + representation

- `board/tile.py` ‚Äî `Tile` enum: the six resource types (sheep, wheat, wood, brick, ore, desert).
- `board/port.py` ‚Äî `Port` enum: the six port types (one per resource plus the 3:1 general port).
- `board/resources.py` ‚Äî Resource constants (`N_PLAYERS`, `N_RESOURCES`, `BANK_INITIAL`), build-cost tuples (`ROAD_COST`, `SETTLEMENT_COST`, `CITY_COST`), jaxtyping aliases for resource arrays, `bank_stock()` (remaining stock of a single resource in one game), and `compute_bank_resources()` which derives remaining bank stock from player holdings.
- `board/dev_cards.py` ‚Äî `DevCard` enum, deck counts (`DEV_CARD_COUNTS`), and `DEV_CARD_COST`. The dev-card *arrays* live on `BoardState` (see state.py).
- `board/layout.py` ‚Äî Static board representation. Defines `BoardLayout` (tile resources, number tokens, port allocation), the module-level jaxtyping aliases for its arrays, and the static `int32` incidence maps generated once by `_generate_mappings()`: `EDGE_V` (edge‚Üíendpoints ‚Äî the COO `edge_index`), `TILE_V` (tile‚Üícorner vertices), and `PORT_V` (port‚Üívertices). These are dense and padding-free ‚Äî the rule modules derive vertex-incidence on the fly by scattering over them (PyG-style; see `mechanics/placement` / `mechanics/trade` / `mechanics/dice`), so no ragged vertex‚Üí* reverse maps or padding sentinels are stored (shape invariants checked in `tests/board/test_layout.py`). `NO_INDEX` survives only as the uint8 "unclaimed award" marker on `BoardState`. `make_layout()` randomly generates a batched `BoardLayout`; its `number_placement` option ("random" default / "spiral") selects how number tokens are laid: "spiral" follows the rulebook's variable set-up ‚Äî the A..R token values (`SPIRAL_NUMBERS`) dealt along a counterclockwise inward spiral of tile indices (`SPIRAL_TILE_ORDER`, host-generated by `_spiral_tile_order()`), skipping the desert, implemented as a vectorized exclusive cumsum-rank + gather so it stays traceable (terrain/ports use the same key stream in both modes, so a seed's terrain is identical across modes; the fixed start corner/handedness is distributionally irrelevant because the terrain shuffle is rotation/reflection-invariant). `desert_tile()` reads the per-lane desert tile index off a `tile_resource` (the robber's rulebook starting position, needed because tile positions are shuffled). Also exposes host-side cube-coordinate ‚Üî index lookups for human-readable board inspection ‚Äî `vertex_index`/`vertex_cube`, `edge_index`/`edge_cubes`, `tile_index`/`tile_cube` (cube `(q, r, s)` triples: tile centres sum to 0, vertices to ¬±1) ‚Äî plain Python, not for use inside a trace; `tests/render.py` reuses these instead of re-deriving the geometry.
- `board/state.py` ‚Äî Mutable game state. Defines `GamePhase`, building-stock/win constants, `BoardState` (board occupancy, dev-card deck/hand/knights, turn-flow flags such as phase/current_player/has_rolled/free_roads, and the Longest Road / Largest Army holders plus a PRNG key), `make_board_state(batch_size, key, n_players)` which initialises a fresh game in the setup phase (`n_players` 2..4, default 4 ‚Äî the per-player arrays are **sized to the seated count**, so a 2-player game carries no padding rows and the rule modules sweep/scatter over fewer player lanes; the `BoardState.n_players` *property* reads the count back off the player axis as a static Python int ‚Äî all games in a batch seat the same number; the jaxtyping `players` dim is symbolic), `to_u8(x)` ‚Äî the saturating uint8 cast through which state-array arithmetic is committed, and `tree_select(mask, a, b)` ‚Äî the per-leaf `where` over two single-game states that the action layer uses for branchless application. Players are 0-indexed except in `vertex_owner`/`edge_road` (player + 1, 0 = empty). Also defines the **single-game (un-batched) jaxtyping aliases** the rule modules annotate with ‚Äî `EdgeRoadVec`/`VertexOwnerVec`/`VertexTypeVec`/`PlayerMaskVec` plus the `IntScalar`/`BoolScalar`/`KeyScalar` scalars (the batch-free counterparts of the batched `*Array` aliases; `PortAllocVec`/`PlayerResourcesVec`/`DevDeckVec` live beside their batched forms in `layout`/`resources`/`dev_cards`).
- `board/__init__.py` ‚Äî Top-level board type `Board = tuple[BoardLayout, BoardState]` (static layout paired with mutable state) plus action-agnostic construction/shaping helpers used by tests and callers (`make_board` ‚Äî which seeds the robber on the desert tile (rulebook) and forwards `n_players`, `replicate`, `set_phase`, `to_main`, `give`, `place_settlement`/`place_road`/`place_city`, `give_dev_card`, `set_robber`). Re-exported at the package root, so `from catan_engine.board import Board, make_board` works.

### `mechanics/` ‚Äî rule helpers + action layer

- **Rule modules** ‚Äî the single-game, **traceable/vmappable** rule helpers, split by domain (all written without `int()`/NumPy/Python branching on traced values; vertex-incidence is computed by **scattering** per-edge/-tile values over the dense `EDGE_V` / `TILE_V` / `PORT_V` maps from `board/layout.py` ‚Äî the JAX analogue of PyG message passing ‚Äî so there are no padding sentinels; batch by wrapping callers in `jax.vmap`). Their public functions annotate array params with the **single-game** jaxtyping aliases above (`EdgeRoadVec`, `IntScalar`, ...), and these annotations are **runtime-enforced during tests**: `tests/conftest.py` installs a jaxtyping import hook (backed by `beartype`, a dev dependency) scoped to exactly these modules, so every call and `jit` trace checks shapes/dtypes. `mechanics/common`, `mechanics/action`, and `env` are excluded ‚Äî they hold functions annotated with the *batched* aliases that run unbatched under `vmap` (the cores in the hooked rule modules, by contrast, are annotated single-game with `IntScalar` / `BoolScalar` / `SingleResources`, so they *are* enforced).
  Each topical module holds its sub-rule helpers **and** the action cores built on them (a core is a single-game traceable transition; each exposes the *batched* public API `<name>_available(board, params)` / `<name>_step(board, params)` ‚Äî module-level `jit(vmap(...))` ‚Äî plus the private single-game `_<name>_avail` / `_<name>_apply` that the dispatch in `action.py` calls ‚Äî `_<name>_apply` takes the precomputed `available` legality as an argument rather than computing it, see the `action.py` bullet). A core applies **only** its core state change and returns `SUCCESS`/`INVALID`; the award reassignment and the win check are **stage 2**, run once per step by `awards.resolve_step` (see the `awards.py` and `action.py` bullets). The few `*_step` wrappers whose action can change an award or win the game (BuildRoad / BuildSettlement / BuildCity / BuyDevelopmentCard / PlayKnight) route their core output through the batched `awards.resolve_step_b` so the standalone API stays fully resolved.
  - `mechanics/placement.py` ‚Äî settlement / road placement legality (`distance_rule_ok`, `settlement_connected`, `road_placeable`, `setup_road_placeable`, via a `_scatter_to_vertices` helper over the edge_index) **and** the `BuildRoad` / `BuildSettlement` / `BuildCity` cores.
  - `mechanics/longest_road.py` ‚Äî the longest-road *length*: the explicit-stack iterative DFS `longest_road_length` via `lax.while_loop`, edge-seeded; fed a sentinel-free CSR adjacency `ADJ_INDPTR`/`ADJ_EDGE`/`ADJ_NBR`/`ADJ_DEG` derived from `EDGE_V` (also imported by `awards.settlement_break_gate`). Each DFS frame is **packed into a single int32**: bits `[0, MAX_ROADS)` hold the used-edge set ‚Äî a running count over the player's edges (`owned_rank`) maps the `<= MAX_ROADS` (15) owned edges to bit positions, so set/test are scalar bit ops (vs a `(STACK_CAP, N_EDGES) bool` stack, ~9x faster batch-100 rollout) ‚Äî and the bits above hold the tip vertex; the trail length is the `population_count` of the used set, needing no field, so the whole stack is one `(STACK_CAP,) int32` array (one gather per pop and one scatter per push instead of three of each). Seeding is **endpoint-only** over the owned edges (compacted to the front via a cumsum over kept directions; `sp <= 2 * n_owned <= 30` rather than `2 * N_EDGES = 144`): a maximal trail's endpoints are odd-owned-degree or impassable vertices, so a direction `u -> v` is seeded only when `u` is such a start, plus a fallback **forward seed for edges with neither endpoint a start** ‚Äî that covers closed trails (rotatable/reflectable to begin anywhere), at the cost of also re-seeding dominated interior edges of even chains (a per-edge test can't tell the two apart; exact either way, ~1.7x on the fused worst-case pop count). Non-owned seeds would only re-explore the same trails and made every lane pay a 144-iteration `while_loop` floor; cutting them is a further ~5x. The hand-built shape cases (loop, lollipop, theta, dumbbell, Y, blocked loop) live in `tests/mechanics/test_rules.py::TestLongestRoadTopologies` with rendered-board snapshots. The loop pops in **blocks of `_POP_K = 32`** (K-way pop): the top `min(_POP_K, sp)` frames are popped and expanded together into a `(_POP_K, MAX_VERTEX_DEGREE)` grid of candidates, pushed contiguously via an exclusive cumsum over the valid flags (invalid ‚Üí `_DUMP`) ‚Äî exact because the DFS is a pure worklist (`best` is a max over popped frames; order is irrelevant), and it divides the `while_loop` trip count by ~K (a further ~6.4x at B=512, ~3.9x at B=64, ~1.6x at B=8, ~1.1x at B=1 on GPU; K=32 beat 8/16/64 in a sweep). `STACK_CAP` is `2*MAX_ROADS + min(2*MAX_ROADS, _POP_K) + (MAX_ROADS-3)*_POP_K + 1` (= 445): seeds + first-block doubling + K net frames per *branchable* depth (a frame pushes ‚â§ deg‚àí1 children because the arrival edge is in the used set, and pushing 2 needs two unused owned edges, so only trail lengths ‚â§ MAX_ROADS‚àí2 branch) + the `_DUMP` scratch slot ‚Äî proven and tight in the abstract stack model, see `docs/longest-road-stack-bound.html` (self-contained page with figures; regenerate via `tools/gen_stack_bound_doc.py`); still fuzzed by `test_rules.py::test_dfs_peak_sp_stays_below_dump` (observed peak 30 ‚â™ 445) because JAX silently *drops* out-of-bounds scatter updates, making the fuzz test the real overflow guard ‚Äî it **relies on the rule invariant `n_owned <= MAX_ROADS`**, which `tests/mechanics/_occupancy.py` therefore enforces when generating random occupancies. The `needed` parameter seeds the DFS empty for a masked-off lane, so it adds no loop iterations under `vmap`.
  - `mechanics/awards.py` ‚Äî award reassignment (the `recompute_*` holders for Longest Road / Largest Army, with the rulebook tie rule in `_reassign_award`) and the **stage-2 step resolver**. `recompute_longest_road` `vmap`s the DFS over the player axis ‚Äî one fused `while_loop` paying a single max trip count instead of `N_PLAYERS` sequential loops each paying its own. `resolve_step(state, result, longest_road_needed)` (and its batched `resolve_step_b`): `recompute_awards` (both holders) then upgrade a `SUCCESS` to `GAME_COMPLETE` if any player hit the win threshold. Factored out of the cores so the expensive Longest Road DFS runs **once** per step, not once per `lax.switch` branch (every branch executes under `vmap`). The DFS is further **gated per lane** (`longest_road_needed`): only a *successful* BuildRoad can extend a length and only a *successful* BuildSettlement can break one (setup placements stay under the 5-road threshold; edges never disappear; a city keeps the vertex owner), tightened by the **build gates** `road_build_gate` (builder owns ‚â• 5 roads ‚Äî a trail's length is bounded by the road count) and `settlement_break_gate` (some single opponent owns ‚â• 2 of the vertex's incident edges ‚Äî one edge only makes the vertex a legal trail *endpoint*). The success-conjoined forms `road_build_needed` / `settlement_break_needed` (+ batched `*_b`) are the single gating helpers that both `apply_action` and the standalone `*_step` wrappers pass to `resolve_step`; every other lane keeps its stored holder/length ‚Äî a gated-off lane's DFS is seeded empty, contributing **zero iterations** to the vmapped `while_loop` (its trip count is the max over lanes, so non-qualifying steps are near-free).
  - `mechanics/trade.py` ‚Äî maritime `port_ratio` (lives here, not `board/port.py`, to avoid the `layout`‚Üí`port` import cycle) **and** the `MaritimeTrade` core.
  - `mechanics/dice.py` ‚Äî `roll_dice` and the `distribute_resources` production payout (with the bank-cap rule) **and** the `RollDice` core.
  - `mechanics/robber.py` ‚Äî the 7-resolution mechanics: `robber_victim_mask` and the random `steal`, the shared `valid_robber_victim` / `apply_steal` (reused by `PlayKnight`), **and** the `MoveRobber` / `Discard` cores. Discard is **one card per action**: the acting discarder (the lowest-indexed player still owing ‚Äî `agent_selection_single`) gives up one card of the chosen resource and its owed count is decremented; the phase advances to MOVE_ROBBER once every owed count is zero. This keeps the choice space flat (one action per resource) instead of the combinatorial whole-hand splits.
  - `mechanics/setup.py` ‚Äî setup turn order (the traceable snake `_setup_player(setup_index, n_players)` ‚Äî `n_players` is per-game state, so the order is computed arithmetically rather than from a static table; the host-side `setup_order(n_players)` states the same order plainly for tests) and the 2nd-settlement `grant_setup_resources` **and** the `SetupSettlement` / `SetupRoad` cores.
  - `mechanics/development.py` ‚Äî dev-card rules `playable_dev` / `draw_dev_card` (separate from `board/dev_cards.py` to avoid the `state`‚Üí`dev_cards` import cycle) **and** the `BuyDevelopmentCard` / `PlayMonopoly` / `PlayYearOfPlenty` / `PlayRoadBuilding` / `PlayKnight` cores (Knight composes `robber`; its Largest Army recompute is deferred to the stage-2 `awards.resolve_step`).
  - `mechanics/turn.py` ‚Äî the `EndTurn` core.
- `mechanics/common.py` ‚Äî the shared vocabulary every core needs (so the rule modules stay leaves importing only `board.*` + `common`, with no cycle through `action`): `ActionResult` (SUCCESS/INVALID/GAME_COMPLETE) and those scalar codes, the MAIN-phase predicates (`main_after_roll` / `main_no_dev_played`), the **economy helpers** (build-cost vectors `ROAD_COST_ARR` etc., `roads_left`, `count_settlements` / `count_cities`, `can_afford`, `pay`, `player_total_vp`), `agent_selection_single`, and the action-layer jaxtyping aliases ‚Äî batched (`Mask`, `ResultCode`, `IndexParam`, `TwoIndexParams`); the cores use the single-game `IntScalar` / `BoolScalar` from `board.state`. (Generic uint8 saturation lives on `state.to_u8`, bank stock on `resources.bank_stock`.)
- `mechanics/action.py` ‚Äî The aggregation / enumeration layer on top of the cores. (1) The unified dispatch: `ActionType` (the 15-action enum; DomesticTrade is intentionally deferred), the packed `ActionParams`, and `apply_action` / `action_available` ‚Äî a single `(action_type, params)` interface, `lax.switch` over the cores so it stays traceable/vmappable. **`apply_action` takes the precomputed `available` legality as an argument** and hands it to the chosen branch; no branch recomputes avail (under `vmap` every branch runs, so an internal avail call would be paid ~15√ó per lane). The caller computes avail **once** ‚Äî via `action_available` (the switch-based avail, exact for any params; used by the functional `env.step` and tests) or from a cached flat sweep (used by `BatchedCatanEnv`). Application is branchless (candidate always computed, then `state.tree_select`-ed against `available`) and **two-stage**: the switch applies the chosen action's core state change, then `awards.resolve_step` recomputes the awards and resolves the win **once** per step (factored out of the per-branch cores for the perf reason given in the `awards.py` bullet). (2) The **flat action table** (`_build_action_table` ‚Üí `_ATYPE`/`_IDX`/`_TARGET`): one enumerated index per concrete move (Discard is one row per resource), fully static ‚Äî no per-lane fill-in. (3) The switch-free **flat legality** (`_flat_available_b`), the `flat_legality` reverse-map lookup (reads one `(action_type, idx, target)` action's bit out of a cached `(B, N_FLAT)` flat sweep ‚Äî how the env gates an action without recomputing avail; non-table actions read False), and the per-index legality enumerations (`_INDEX_MASKS`), which call each core directly over its own static slice of the parameter space rather than routing the table through the `lax.switch` (which `vmap` would lower to evaluating all 15 branches per entry). (The old per-action-type sweep `_action_type_mask_b` is gone ‚Äî the env reduces that mask straight from the cached flat legality.) Re-exports `ActionResult` / `Mask` / `ResultCode` / `player_total_vp` from `common`. Per-action tests live in `tests/mechanics/actions/` (they call the functional `<name>_step` / `<name>_available`; pytest fixtures in `tests/mechanics/actions/conftest.py`; the shared ASCII `BoardRenderer` lives in `tests/render.py`). Seed expecttest inline expectations that will hold board renders with a **raw** literal (`r""""""`): `EXPECTTEST_ACCEPT=1` preserves the seed's rawness, and a non-raw seed gets every `\` escaped to `\\` (an unreadable mess for hex art); large multi-literal accept runs may need a second pass.

**Engine path:** the engine is fully JAX-native (the `mechanics/` rule modules + `mechanics/action.py`); the old NumPy single-game path has been removed. The trusted differential oracle is the standalone **`catan-reference`** package (a plain-Python, rulebook-faithful implementation), bridged to the engine by `tests/conversion.py` ‚Äî which converts a batched engine `Board` into a list of reference `Game`s (positions translated by cube coordinate) and exposes the per-rule wrappers that `tests/mechanics/test_rules.py` (and friends) check against. `tests/test_reference_equivalence.py` drives both engines with the same action stream and asserts full-state agreement across random games. None of this is part of the shipped package.

### `env/` ‚Äî RL surface

- `env/batched.py` ‚Äî RL environment core. Two layers: (1) the functional batched `step(board, action_type, params)` / `available(...)` ‚Äî a single `(ActionType, ActionParams)` interface over all 15 actions; `step` computes the per-action legality once via the switch-based `available(...)` (so it stays self-validating for arbitrary actions ‚Äî used by tests / the reference oracle) and hands it to `apply_action`. (2) `BatchedCatanEnv`, a stateful **batched PettingZoo-AEC** env over that core (batch axis = parallel games; the acting agent in each lane is its `current_player`, or the next owing player during DISCARD). **A single cached legality source, `self._avail` (`(B, N_FLAT)`):** the switch-free flat sweep (`_flat_available_b`) for each lane's acting player, computed once at `reset` and refreshed at the end of every `step`. `random_actions`, `action_mask`, and the per-step gate all read it instead of recomputing avail. The whole step ‚Äî gate the chosen action via `flat_legality(self._avail, ‚Ä¶)`, apply the avail-free core, score reward / termination, auto-reset finished lanes, refresh `self._vps` and `self._avail` ‚Äî is **one fused `jit` dispatch** (`_env_step_core`, with `batch_size`/`reward_mode`/`auto_reset`/`number_placement`/`n_players` static and the PRNG key threaded); collapsing ~5 kernels into 1 is what makes the rollout faster (small batches are dispatch-bound). `n_players` (2..4, constructor arg, same for every lane) sizes `possible_agents` and is forwarded to `make_board_state` for the initial boards and auto-reset replacements; the per-player arrays (state, observations, rewards) are sized to it, while the flat action table keeps the full 4-player victim domain (rows naming unseated victims are simply never legal, so the flat action space is constant across player counts). AEC surface adapted to batches: `agent_selection` is a `(B,)` array; `rewards`/`terminations`/`truncations` are `(B, n_players)` arrays; `infos` is a single dict of batched arrays carrying `"action_mask"`. `observe(agent)` gives a **partial** view (own hand/dev cards in full, public counts only for opponents); `last()` returns the acting agents' slices. Reward is `"sparse"` (win/loss) or `"vp_delta"` (configurable); `number_placement` ("random"/"spiral") is forwarded to `make_layout` for both the initial boards and auto-reset replacements; terminated lanes **auto-reset** to fresh boards by default (`auto_reset=False` freezes them instead ‚Äî used by the single-game AEC wrapper in `env/aec.py`; the reset branch is a device-side `lax.cond` on `any(done)`, not a host `bool(...)`, avoiding a per-step device‚Üíhost sync while only paying board generation when a lane actually finished). `action_mask()` reduces the `(B, N_ACTION_TYPES)` per-type mask straight from `self._avail` (OR each type's flat rows); `available_indices(action_type)` delegates to the `_INDEX_MASKS` (`(B, D)` sweep over an index-parameterized action's primary domain) enumeration in `mechanics/action.py`. `random_actions(key)` samples a uniformly-random *legal* action per lane (read `self._avail`, then a per-lane masked argmax) ‚Äî the random-rollout driver. The **flat action table** and switch-free **flat legality** both live in `mechanics/action.py` now (see that bullet); this module imports them. Spaces are described with dependency-free `Discrete`/`Box` descriptors. Re-exports `ActionType`, `ActionParams`, `ActionResult`, `N_ACTION_TYPES`. **Public flat-action seam** (what `catan-agents` policies consume): `N_FLAT`, `flat_to_action(flat)` (table gather ‚Üí `(action_type, ActionParams)`), `BatchedCatanEnv.flat_mask()` (the cached `self._avail`), and `flat_available(board)` (the same switch-free sweep as a pure function over any `(layout, state)` ‚Äî what catan-agents' MCTS uses for in-tree masks).
- `env/__init__.py` ‚Äî re-exports `env/batched.py`'s public API so `catan_engine.env.step`, `catan_engine.env.BatchedCatanEnv`, etc. resolve at the package root as before.
- `env/aec.py` ‚Äî Single-game **PettingZoo-AEC** wrapper `CatanAECEnv` (and an `env()` constructor) around `BatchedCatanEnv(batch_size=1, auto_reset=False)`; `n_players` (2..4) is forwarded and sets the agent list. Inherits `pettingzoo.utils.env.AECEnv`: turn-at-a-time `reset`/`step(action)`/`observe`/`last`, per-agent `rewards`/`terminations`/`truncations`/`infos` dicts, and gymnasium spaces. The action space is a single flat `Discrete` enumerating every concrete move (the flat action table ‚Äî `_build_action_table` ‚Äî lives in `mechanics/action.py` and is imported here; flat index ‚Üí `(ActionType, idx, target)`; discarding is one card per action, one flat action per resource, repeating until the owed count is zero); legality is exposed PettingZoo-style as `observation["action_mask"]` (read from the batched env's cached flat legality `self._avail`, refreshed each step ‚Äî shared with `random_actions`). Observations are the partial view from `BatchedCatanEnv.observe`, wrapped `{"observation": {...}, "action_mask": ...}`. Game-over terminates all agents (then the base class dead-steps them out). Needs the optional `rl` extra (`pettingzoo`, `gymnasium`); validated by `tests/env/test_aec.py` (incl. PettingZoo's `api_test`).

### Top level

- `__init__.py` ‚Äî Package entry point (`main`).
